{\Large\bfseries ABSTRACT}
\end{center}
\addcontentsline{toc}{chapter}{Abstract}
\vspace{0.5cm}

IoT devices are everywhere today — in hospitals, factories, homes, and city
infrastructure. But most of them were never designed with security in mind, and
attackers have taken full advantage of that. The bigger problem, though, is not just
the attacks we already know about. It is the ones we have never seen before — the
so-called zero-day threats — that pose the greatest risk to any detection system built
purely on historical patterns.

\vspace{0.5cm}
\noindent This project grew out of a simple question: can we build a single system that
handles both familiar attacks and completely new ones at the same time? Our answer is a
\textbf{hybrid intrusion detection system (IDS)} that pairs a gradient-boosted tree
classifier with a neural-network-based anomaly detector, then fuses their decisions
intelligently. We trained the \textbf{LightGBM} classifier on ten known attack classes
from the Edge-IIoTset dataset~\cite{dataset_edge_iiotset} and got 99.99\% accuracy on
those. Separately, we trained a symmetric \textbf{Autoencoder} on nothing but normal
traffic, so it learns what ``ordinary'' IoT data looks like and raises an alarm whenever
something deviates from that picture. On its own, the Autoencoder reached 99.19\%
accuracy across the full dataset.

\vspace{0.5cm}
\noindent The real test came when we put both models together in a \textbf{Fusion Model}
and threw five attack categories at it that neither model had ever seen during training.
The combined system managed 96.82\% accuracy — a nearly 48-percentage-point jump over a
standalone Isolation Forest baseline that barely scraped 49\%. These results gave us
confidence that mixing supervised and unsupervised methods is not just theoretically
appealing but practically effective.

\vspace{0.5cm}
\noindent \textbf{Keywords:} IoT Security, Intrusion Detection, Zero-Day Attacks,
LightGBM, Autoencoder, Anomaly Detection, Fusion Model, Edge-IIoTset.

\chapter{Introduction}

\section{Background}

When we started thinking about what to build for our final-year project, the
topic of IoT security kept coming up — partly because it is genuinely unsolved,
and partly because the scale of the problem is hard to ignore. According to
recent industry figures, there are now over 15 billion connected IoT devices in
active use, and that number is expected to climb past 30 billion before the end
of the decade~\cite{iot_growth}. These devices show up in almost every corner of
modern life: patient monitors in hospitals, sensors on factory floors, smart
meters in homes, cameras on street corners, and dozens of other places where
reliable, secure operation actually matters.

\noindent The trouble is that most of these devices were built to be cheap and
convenient, not secure. They often run on hardware with very limited processing
power and memory, ship with firmware that rarely gets updated, and are connected
to networks with little or no encryption. Attackers discovered this long ago.
The Mirai botnet incident back in 2016 — where hundreds of thousands of ordinary
home routers and cameras were silently hijacked and used to knock major websites
offline — was a wake-up call that the IoT security problem is not just
theoretical~\cite{mirai_attack}. Since then, things have only grown more complex
as more devices, more protocols, and more attack vectors have appeared
~\cite{iot_security_challenges}.

\noindent Traditional intrusion detection systems try to tackle this by
maintaining a catalogue of known attack signatures. When traffic matches a
pattern in that catalogue, an alert fires. This works reasonably well for attacks
that have been seen and catalogued before, but it completely misses anything new.
Zero-day attacks — those that exploit fresh vulnerabilities or use techniques not
yet in any signature database — sail straight through. That gap is what this
project tries to close.

\section{Problem Statement}

The core challenge we set out to solve is straightforward to describe but
difficult to implement well. Any practical IoT intrusion detection system needs
to do two things simultaneously: it needs to correctly identify the attacks it
has been trained on, and it also needs to flag traffic that does not look like
anything it has seen before. Most existing systems do one or the other, rarely
both.

\noindent More specifically, the three problems we focused on are:
\begin{enumerate}
  \item \textbf{Known attack classification}: The system must accurately classify
        network traffic among multiple known attack categories in real time.
  \item \textbf{Zero-day detection}: The system must identify anomalous traffic
        that deviates from learned normal patterns, even without prior labelled
        examples of the specific attack.
  \item \textbf{Integration}: Both capabilities must be integrated into a single
        unified pipeline rather than operating as disconnected tools.
\end{enumerate}

\section{Objectives}

Our objectives for this project were:
\begin{itemize}
  \item Design a hybrid detection framework that brings together a supervised
        classifier and an unsupervised anomaly detector under one roof.
  \item Evaluate LightGBM as the supervised component and understand where it
        excels and where it falls short.
  \item Train an Autoencoder on clean traffic only and use reconstruction error
        as a way of spotting anomalies, including zero-day attacks.
  \item Build the Fusion Model — the glue that decides which component to trust
        for a given traffic sample — and measure whether combining the two
        models actually helps in practice.
  \item Compare the final system honestly against a simpler baseline so we can
        quantify how much value the hybrid approach really adds.
\end{itemize}

\section{Scope of Work}

We used the publicly available Edge-IIoTset dataset~\cite{dataset_edge_iiotset}
throughout this project, which gave us a realistic, labelled collection of IoT
network traffic spanning fifteen attack categories. Within that scope, our work
covered:
\begin{itemize}
  \item Full preprocessing of the raw dataset, including cleaning, encoding,
        and fitting a feature scaler specifically on normal traffic.
  \item Training and evaluating a LightGBM model on the ten known attack classes.
  \item Training the Autoencoder on normal traffic alone and calibrating its
        anomaly threshold.
  \item Implementing the confidence-based fusion logic that ties the two models together.
  \item Running a thorough evaluation with classification reports, confusion matrices,
        and a direct comparison against Isolation Forest as a baseline.
\end{itemize}

\noindent What we did not attempt: real-time packet capture, deployment on actual
IoT hardware, or identifying \textit{which} specific zero-day attack type has occurred
(the system only flags that something is wrong). Those are natural next steps, but
they were beyond the time and resource constraints of a final-year project.

\section{Report Organisation}

The rest of this report is laid out as follows. Chapter~2 reviews the relevant
literature on IoT security, intrusion detection, and the machine learning techniques
we drew on. Chapter~3 describes how we designed the system — the dataset, preprocessing
steps, and the architecture of each component. Chapter~4 walks through the actual
implementation in enough detail that someone familiar with Python could reproduce our
work. Chapter~5 presents and discusses the experimental results. Chapter~6 steps back
to interpret those results, acknowledge limitations, and suggest where future work
could go. Chapter~7 wraps up with our conclusions.

\chapter{Literature Review}

\section{The Growing IoT Security Problem}

Understanding why IoT security is hard requires looking at how these devices
actually get attacked. Pa et al.~\cite{iot_malware} showed that even simple malware
like Mirai can grow into a massive botnet almost effortlessly, because a huge
proportion of IoT devices ship with the same default usernames and passwords and
owners rarely change them. Their honeypot experiments captured hundreds of thousands
of login attempts per day targeting common IoT credentials — a pattern that has only
intensified since their study was published. The 2016 Mirai attack on DNS provider
Dyn~\cite{mirai_attack} demonstrated what can happen when those compromised devices
are weaponised: sustained traffic floods that brought down well-known services for
hours across the east coast of the United States.

\noindent Network intrusion detection as a field has been around for decades, and the
fundamental choices have not changed much. Liao et al.~\cite{nids_survey} described
the landscape clearly: you can either keep a database of known attack fingerprints
and match incoming traffic against them, or you can model what normal behaviour looks
like and treat everything that deviates significantly as suspicious. The first approach
(signature-based) is precise and fast but completely useless against anything novel.
The second approach (anomaly-based) can in principle catch things no one has labelled
yet, but it tends to produce more false alarms because ``unusual'' does not always mean
``malicious.'' Neither approach alone is good enough for a real deployment, which is
why hybrid systems have attracted so much research interest.

\section{Machine Learning Applied to Intrusion Detection}

Early machine learning work on intrusion detection mostly used the KDD Cup 1999
dataset~\cite{kdd_cup}, which, despite being synthetic and old, served as a common
benchmark for many years. Decision trees and random forests showed strong results on
it, and that encouraged researchers to keep pushing in the ML direction even as the
dataset's limitations became more apparent.

\noindent The bigger shift came with gradient boosting. Among the various gradient
boosting libraries, LightGBM~\cite{lightgbm_paper} — released by Microsoft Research in
2017 — stood out for its speed and memory efficiency. Instead of sorting data at each
split point (which is expensive), it bins continuous features into histograms first and
then works with those bins. It also grows trees leaf-by-leaf rather than level-by-level,
which tends to give better accuracy for the same number of leaves. For network traffic
classification, where datasets easily reach hundreds of thousands of rows and dozens of
features, these properties matter a lot. Zhang et al.~\cite{lgbm_ids} confirmed this
in a more recent study, showing that LightGBM with carefully selected features can
outperform both older ensemble methods and many deep learning approaches on network
intrusion data.

\section{Autoencoders as Anomaly Detectors}

An autoencoder is a neural network with a bottleneck in the middle: the first half
compresses the input down to a smaller internal representation, and the second half
tries to reconstruct the original from that compressed form~\cite{autoencoder_original}.
When you train one entirely on examples of one class — say, normal traffic — it gets
very good at compressing and reconstructing that class. Present it with something
different, like an attack packet pattern, and the reconstruction is noticeably worse.
The difference between the input and the reconstruction, measured as mean squared
error, becomes a practical anomaly score.

\noindent This approach has shown up in several detection domains beyond networking.
Pumsirirat and Yan~\cite{ae_fraud} used autoencoders for credit card fraud detection
and demonstrated that the reconstruction-error approach generalises well even when
fraud examples are extremely rare in training. Baur et al.~\cite{ae_medical} applied a
similar idea to brain MRI scans — training only on healthy images and flagging
reconstructions that diverge significantly as potential pathologies. For network
intrusion specifically, Shone et al.~\cite{ae_ids} showed that a non-symmetric
autoencoder architecture could outperform several classical algorithms on the NSL-KDD
dataset, though they noted that threshold selection remains a sensitive design choice.

\noindent Some researchers have pushed further by replacing the standard autoencoder
with generative adversarial networks. Schlegl et al.~\cite{ganomaly} argued that a GAN
discriminator provides a richer anomaly signal than simple reconstruction error.
Zenati et al.~\cite{efficient_gan} found similar benefits but noted that GANs are
considerably harder to train stably on tabular data. For our purposes — a structured
feature matrix from a network capture — a straightforward autoencoder offered a better
balance of reliability, interpretability, and training speed.

\section{Hybrid Detection Systems}

The case for combining supervised and unsupervised methods is fairly intuitive:
give the supervised model responsibility for attacks it knows well, and let the
anomaly detector handle anything unfamiliar. What is less obvious is how to merge
their decisions cleanly. Yin et al.~\cite{hybrid_ids} explored pairing recurrent neural
networks with anomaly-based components and found that the combined system was more
robust than either alone, particularly on traffic patterns that had evolved slightly
since training. Khraisat et al.~\cite{hybrid_survey} surveyed a wide range of hybrid
IDS architectures and concluded that multi-model approaches consistently perform better
than single-model ones, though they also pointed out that the fusion mechanism itself
is a major source of variation between systems — a finding that directly shaped how we
designed our Fusion Model.

\noindent Our own approach to fusion is a confidence-based router: the supervised
model gets first say, and only when it lacks confidence does the anomaly detector
step in. This keeps the Autoencoder out of the critical path for traffic the LightGBM
already handles well, which reduces unnecessary false positives while still providing
zero-day coverage.

\section{Datasets for IoT Intrusion Detection Research}

One frustration that kept coming up in our literature search was the shortage of
good IoT-specific datasets. The KDD Cup data is ancient. NSL-KDD is cleaner but
still not representative of modern IoT protocols. CICIDS datasets are better but
focus on general enterprise traffic rather than constrained IoT devices.

\noindent The Edge-IIoTset dataset~\cite{dataset_edge_iiotset}, released by Ferrag et al. in 2022, addressed many of these gaps by providing a comprehensive benchmark from a realistic IoT testbed with fourteen attack categories. We describe this dataset in detail in Chapter~3, as it forms the basis of all our experiments.

\section{Research Gap}

From our review of existing work, we identified a clear gap: while supervised classifiers like LightGBM achieve near-perfect accuracy on known IoT attacks, they offer no mechanism for detecting novel threats. Conversely, unsupervised anomaly detectors like autoencoders can flag unknown traffic but lack the precision to distinguish between specific attack types. No prior work on the Edge-IIoTset dataset has attempted a confidence-based fusion strategy that dynamically routes traffic between a supervised classifier and an unsupervised anomaly detector based on prediction confidence. This project addresses that gap by proposing and evaluating exactly such a hybrid system.

\chapter{System Design}

\section{Overall Architecture}

Before writing a single line of code, we spent time sketching out what the
system actually needed to do. The requirements pulled us in two directions at
once: high accuracy on known attack types (which pushes toward a supervised,
labelled-data approach) and the ability to detect things we had never trained on
(which pushes toward an unsupervised approach). Trying to satisfy both with one
model seemed like a bad idea — a classifier trained on ten attack classes has no
vocabulary for an eleventh one it has never encountered. So we went with three
separate, cooperating components instead.

\noindent The first is a \textbf{Preprocessing Module} that cleans the raw
network capture data and transforms it into a consistent numerical feature matrix.
The second is a \textbf{LightGBM Classifier} that handles the ten attack types it
was trained on. The third is an \textbf{Autoencoder Anomaly Detector} trained
exclusively on clean, normal traffic — it never sees a labelled attack during
training, so anything that diverges from what it learned to reconstruct stands out.
Tying it all together is the \textbf{Fusion Model}, which decides on a per-sample
basis whether to trust the LightGBM result directly or to defer to the Autoencoder
for a second opinion.

\noindent The flow through the pipeline looks like this: raw traffic enters the
preprocessor, scaled features feed into both models in parallel, and the Fusion
Model produces the final binary verdict — Normal or Attack.

\section{Dataset Description}

\subsection{Edge-IIoTset}

We chose the Edge-IIoTset dataset~\cite{dataset_edge_iiotset} because it is one
of the few publicly available IoT traffic collections built from a real physical
testbed rather than simulated traffic. Ferrag et al. set up ten distinct IoT device
types — including IP cameras, temperature and humidity sensors, flame detectors,
water quality monitors, and smart home controllers — and recorded their network
activity under both normal operation and fourteen different attack scenarios. The
captures went through Wireshark, so the feature set includes actual packet-level
fields from protocols like MQTT, Modbus TCP, HTTP, DNS, and ARP that are genuinely
used in IoT deployments.

\subsubsection{Dataset Statistics}

\subsection{Data Split Strategy}

We ended up with two completely separate partitions:

\begin{itemize}
  \item \textbf{Known training set}: 121,293 samples covering Normal traffic
        plus the nine known attack types. LightGBM trains on 80\% of this
        (97,034 samples) and is evaluated on the remaining 20\% (24,259 samples).
  \item \textbf{Zero-day test set}: 31,096 samples from the five held-back
        attack categories, combined with 24,152 normal traffic samples to form
        a 55,248-sample evaluation set for the Fusion Model.
\end{itemize}

\noindent The discipline of keeping the zero-day categories completely out of
training is important. We were careful that even the StandardScaler was fitted
only on the normal traffic subset, not on any attack data — including the known
attacks — to avoid any leakage of attack statistics into the Autoencoder's
normalisation baseline.

\section{Preprocessing Pipeline}

The raw dataset needed a fair amount of work before it was usable. Some of the
issues were straightforward (duplicate rows, missing values) and others were
more specific to network packet data (high-cardinality string columns like HTTP
URIs and IP addresses that encode no general pattern).

\begin{enumerate}
  \item \textbf{Column pruning}: We dropped ten columns that were either
        identifiers (IP addresses, frame timestamps) or free-text fields
        (HTTP URIs, file data) that would cause data leakage or could not be
        meaningfully encoded.
  \item \textbf{Deduplication and null removal}: After dropping duplicates and
        rows with missing values, the dataset shrank from 157,800 to 152,389
        rows. That is about a 3.4\% reduction — not large, but worth doing.
  \item \textbf{Categorical encoding}: Columns like TCP flags, HTTP request
        method, and MQTT topic names were string-typed. We applied
        \texttt{LabelEncoder} to convert them to integers so the models could
        consume them.
  \item \textbf{Datatype casting}: All feature columns were cast to
        \texttt{float32} or \texttt{int32}. This halved memory usage compared
        to the default \texttt{float64}, which mattered when loading the full
        dataset in Colab.
  \item \textbf{Feature scaling}: A \texttt{StandardScaler} was fitted on the
        normal traffic rows only and then applied to scale features for the
        Autoencoder. LightGBM does not need scaling (tree models are invariant
        to monotonic feature transformations), so unscaled features were passed
        to it directly.
\end{enumerate}

\section{Autoencoder Design}

Our Autoencoder is a fully connected, symmetric network with a three-layer encoder
and a mirrored three-layer decoder (seven layers total including input and output):

\begin{table}[H]
\centering
\caption{Autoencoder layer-by-layer architecture}
\label{tab:ae_arch}
\begin{tabular}{cll}
\toprule
\textbf{Layer} & \textbf{Units} & \textbf{Activation} \\
\midrule
Input           & 51  & --- \\
Encoder Dense 1 & 64  & ReLU \\
Encoder Dense 2 & 32  & ReLU \\
Bottleneck      & 16  & ReLU \\
Decoder Dense 1 & 32  & ReLU \\
Decoder Dense 2 & 64  & ReLU \\
Output          & 51  & Linear \\
\midrule
\multicolumn{2}{l}{Total trainable parameters} & 11,907 \\
\bottomrule
\end{tabular}
\end{table}

\noindent The bottleneck squeezes 51 input features down to a 16-dimensional
representation — roughly a 3:1 compression ratio. We chose 16 units after
experimenting with 8 and 32: 8 produced noticeably higher reconstruction errors
on normal traffic (the bottleneck was too tight), while 32 gave no improvement
over 16 but trained more slowly.

\noindent Training details:
\begin{itemize}
  \item Input: 24,152 normal traffic samples, split 80/20 into training
        (19,321) and validation (4,831) sets.
  \item Loss: Mean Squared Error between input and reconstructed output.
  \item Optimiser: Adam with default learning rate (0.001).
  \item Batch size: 256. Training ran for up to 50 epochs but was stopped
        early at epoch 45 when validation loss stopped improving for five
        consecutive epochs.
\end{itemize}

\noindent \textbf{Threshold calibration}: After training, we ran the 4,831
validation samples through the model and recorded every sample's reconstruction
error. We then set the anomaly threshold $\tau$ at the 95th percentile of those
errors:

\begin{equation}
  \tau = \text{Percentile}_{95}\!\left(\left\{\,\|x_i - \hat{x}_i\|^2
  \;\middle|\; x_i \in X_{\text{val}}\right\}\right) = 0.01218
\end{equation}

\noindent Choosing the 95th percentile means we accept up to 5\% of normal
traffic being flagged as anomalous (false positives) in exchange for higher
sensitivity on actual attacks. We also tested the 90th and 99th percentiles;
the 95th offered the best trade-off on the held-out normal validation set.

\section{Fusion Model Design}

The Fusion Model does not contain any trainable parameters — it is purely a
routing mechanism built on top of the two trained models. The logic is as follows:

\begin{algorithm}[H]
\caption{Fusion Model Decision Logic}
\label{alg:fusion}
\begin{algorithmic}[1]
\Require Input traffic sample $x$
\Ensure Classification label $y \in \{0=\text{Normal},\; 1=\text{Attack}\}$
\State Compute LightGBM class probabilities: $p \leftarrow \text{LightGBM.predict\_proba}(x)$
\State $p_{\text{normal}} \leftarrow p\,[\text{index of Normal class}]$
\If{$p_{\text{normal}} \geq 0.90$}
    \State $y \leftarrow 0$ \quad \textit{// LightGBM is highly confident this is normal}
\Else
    \State Compute reconstruction error: $e \leftarrow \text{MSE}(x,\; \text{Autoencoder}(x))$
    \If{$e > \tau$}
        \State $y \leftarrow 1$ \quad \textit{// Autoencoder flags it as anomalous}
    \Else
        \State $y \leftarrow 0$ \quad \textit{// Autoencoder also considers it normal}
    \EndIf
\EndIf
\Return $y$
\end{algorithmic}
\end{algorithm}

\noindent The 90\% confidence threshold was chosen empirically. We ran the Fusion
Model with thresholds of 0.70, 0.80, 0.90, and 0.95, and found that 0.90 produced
the best balance between minimising missed attacks (false negatives) and keeping
false alarms (false positives) at an acceptable level. Below 0.90, too many
borderline-normal samples got routed to the Autoencoder unnecessarily, inflating
false positives. Above 0.90, a small number of subtle zero-day samples slipped
through without being checked by the Autoencoder.

\noindent One practical consequence of this design: the Autoencoder only runs on
samples where LightGBM is uncertain. Since LightGBM is highly confident on most
traffic, the Autoencoder handles only a minority of samples, which keeps overall
inference time manageable.

\section{Autoencoder Training}

The Autoencoder training took slightly more care than the LightGBM step because
we had to make sure the model never saw any attack traffic — not even the known
attacks — during fitting. We also had to be deliberate about when to stop training,
since running too many epochs on such a small normal-traffic dataset can cause the
model to become overly specialised and start producing poor reconstructions on
slightly varied normal samples.

\begin{lstlisting}[language=Python, caption={Autoencoder architecture and training from Notebook 03}]
from tensorflow.keras import layers, models, callbacks

\section{Fusion Model Implementation}

The \texttt{fusion\_predict} function below is the heart of the combined system.
It iterates over samples, asks LightGBM for class probabilities first, and only
invokes the Autoencoder when LightGBM's confidence in Normal falls below the 90\%
threshold. In practice, well over 90\% of samples are handled by LightGBM alone.

\begin{lstlisting}[language=Python, caption={Fusion Model prediction logic from Notebook 04}]
def fusion_predict(X_scaled, X_raw):
    """
    X_scaled : scaled features (51 dims) for the Autoencoder
    X_raw    : unscaled features for LightGBM
    Returns  : numpy array of binary labels (0=Normal, 1=Attack)
    """
    predictions = []
    lgb_probs = clf.predict_proba(X_raw)

    for i in range(len(X_raw)):
        normal_conf = lgb_probs[i, class_to_idx['Normal']]

        if normal_conf >= 0.90:
            # LightGBM is confident enough -- trust it directly
            predictions.append(0)
        else:
            # LightGBM is uncertain; defer to the Autoencoder
            sample = X_scaled[i].reshape(1, -1)
            recon  = autoencoder.predict(sample, verbose=0)
            error  = np.mean(np.square(sample - recon))
            predictions.append(1 if error > threshold else 0)

    return np.array(predictions)
\end{lstlisting}

All models were evaluated on held-out test sets that were never seen during training. No hyperparameter was tuned using the final test numbers. The following metrics were used throughout:
\begin{itemize}
  \item \textbf{Accuracy}: Overall fraction of correctly classified samples.

\chapter{Discussion}

\section{What the Results Actually Tell Us}

\subsection{The Hybrid Approach Holds Up}

The central argument of this project was that neither a purely supervised nor a
purely unsupervised system is sufficient on its own for IoT intrusion detection
in the presence of zero-day threats. The results support that argument fairly
clearly. LightGBM is an exceptional classifier for the attacks it was trained on
--- 99.99\% accuracy is hard to argue with --- but feed it a Ransomware or XSS
packet and it will assign it to whichever known class it most resembles. It has
no mechanism for saying ``this looks like nothing I have been taught.'' The
Autoencoder, on the other hand, is indifferent to attack labels altogether. It
just asks whether something looks like normal traffic or not, and on that basis
it caught every single attack in the dataset. The problem with relying on the
Autoencoder alone is that its 5.13\% false positive rate would generate a
constant stream of spurious alerts in a real deployment.

\noindent The Fusion Model combines the precision of the classifier with the
open-ended detection capability of the anomaly detector, and the 96.82\% result
on completely unseen attack categories is the practical payoff. It is not quite
as clean as the standalone LightGBM number, but it is operating on a fundamentally
harder problem --- detecting things it was never shown.

\subsection{Why the 90\% Confidence Threshold Worked}

We spent longer than expected settling on the 90\% threshold for routing decisions.
Our initial instinct was to use 0.95, reasoning that a higher bar for the supervised
model would push more uncertain samples to the Autoencoder and improve zero-day
recall. In practice, this did improve recall very slightly but also increased false
positives noticeably, because many borderline normal samples that LightGBM handled
correctly at 0.90 were unnecessarily re-evaluated by the Autoencoder at 0.95 and
sometimes flagged as anomalous. Going below 0.90 to 0.80 had the opposite problem:
LightGBM was so rarely overruled that the Autoencoder was almost never used, and
zero-day recall dropped. The 0.90 threshold sat at a sweet spot where LightGBM
retained its dominance over known traffic while still deferring on genuinely
ambiguous cases.

\subsection{The 520 Missed Zero-Day Attacks}

We paid particular attention to the 520 false negatives --- the zero-day attack
samples that the Fusion Model passed as normal. These came overwhelmingly from
the Fingerprinting and Port Scanning categories. Both of these attack types share
a characteristic that makes them hard to catch with a reconstruction-error
approach: they are low-intensity and slow. A port scan sends a small number of
probing packets spread over time; each individual packet may be nearly
indistinguishable from a routine connection attempt. The Autoencoder sees one
sample at a time and has no temporal memory, so it cannot recognise the pattern
across a sequence of packets. This is probably the most significant structural
limitation of the current system.

\section{Limitations We Need to Acknowledge}

We want to be honest about what this system cannot do and where the evaluation
setup was favourable to our results:

\begin{enumerate}
  \item \textbf{Binary output only.} The Fusion Model tells you whether something
        is an attack or not. It cannot tell you \textit{which} kind of zero-day
        attack it is. In a real SOC environment, knowing the attack category matters
        for triage and response. This is the most immediately useful extension of
        the current work.

  \item \textbf{Sample-by-sample evaluation.} We evaluated the system one packet
        feature-vector at a time. Real network traffic arrives as a stream, and
        some attacks (like port scanning) only become detectable when you look at
        sequences or aggregate statistics over a time window. Our results may
        overestimate performance on slow, low-volume attack types for exactly this
        reason.

  \item \textbf{Static anomaly threshold.} The value $\tau = 0.01218$ was fixed
        after training and never updated. In a production system, what counts as
        ``normal'' drifts over time as devices are added, firmware is updated, or
        usage patterns change. A static threshold will gradually become stale.

  \item \textbf{Hardcoded file paths.} All notebooks reference absolute Google
        Drive paths that only work on our Colab accounts. Anyone trying to
        reproduce the experiments needs to update these paths manually, which is
        a minor but unnecessary friction.

  \item \textbf{Retained zero-importance features.} The 14 features with zero
        LightGBM importance are dead weight. They bloat the feature matrix fed
        to the Autoencoder and could potentially add noise to its reconstructions.
        We kept them for consistency with the trained models, but removing them
        is an easy win for future versions.

  \item \textbf{Class imbalance.} Several attack categories have significantly
        fewer samples than others. The weighted average metrics in our reports
        largely hide this imbalance. A more rigorous evaluation would report
        per-class metrics with equal weight rather than weighted averages.
\end{enumerate}

\section{How Our Results Sit Against Related Work}

The authors of the Edge-IIoTset dataset~\cite{dataset_edge_iiotset} themselves
benchmarked several standard algorithms on their data, reporting accuracy figures
in the 95--98\% range for known attack classification. Our LightGBM result of
99.99\% exceeds their best reported number, which we attribute partly to the
feature engineering step and partly to LightGBM's strength on structured tabular
data compared to the algorithms they tested.

\noindent Khraisat et al.~\cite{hybrid_survey}, in their survey of hybrid IDS
systems, found that the top-performing combined systems achieved around 97--99\%
accuracy on datasets that mixed known and unknown traffic. Our Fusion Model sits
at 96.82\%, which is within that competitive range despite operating on a strictly
zero-shot evaluation --- the five test categories were never accessible in any
form during training, which is a stronger test than most hybrid IDS papers apply.

\noindent The Isolation Forest result at 49\% is consistent with findings by
Emmott et al.~\cite{iforest_limits}, who showed that unsupervised anomaly detectors
perform poorly when the majority of data points are anomalous (which is the case
here, where attacks vastly outnumber normal samples) and when the anomaly distribution
is multimodal and spread across many different cluster shapes.

\section{Directions for Future Work}

Working through this project opened up several questions we did not have time to
answer properly:

\begin{enumerate}
  \item \textbf{Zero-day attack categorisation.} Once the Fusion Model flags a
        sample as a zero-day attack, can a second-stage clustering or classification
        step assign it to an attack family? Semi-supervised approaches that group
        anomalies by similarity could be a practical starting point.

  \item \textbf{Sequence-aware detection.} Replacing the per-sample evaluation
        with a sliding-window or LSTM-based approach would give the Autoencoder
        access to temporal context, which should help with slow-scan attacks like
        Port Scanning and Fingerprinting.

  \item \textbf{Adaptive threshold.} A threshold that updates periodically based
        on recent normal traffic — perhaps using a rolling percentile over a
        buffer of recent normal samples — would keep the system calibrated as
        traffic patterns evolve.

  \item \textbf{Variational Autoencoder.} Replacing our deterministic Autoencoder
        with a VAE would give a probabilistic latent representation. Anomaly
        scoring via the reconstruction probability (rather than raw MSE) is more
        principled and tends to be better calibrated, particularly for
        high-variance normal traffic.

  \item \textbf{Feature pruning.} Removing the 14 zero-importance features and
        retraining both models would reduce dimensionality without sacrificing
        accuracy and might actually improve Autoencoder generalisation slightly.

  \item \textbf{Edge deployment.} Profiling and quantising the models for
        deployment on a Raspberry Pi or similar IoT gateway hardware would
        turn this from an academic exercise into something closer to a real
        product. LightGBM is already very lightweight; the Autoencoder with
        only 11,907 parameters should be deployable with minimal optimisation.
\end{enumerate}

\chapter{Conclusion}

We set out to answer one question: can a single system reliably detect both known
and completely unseen IoT network attacks? Based on our experiments, the answer
is yes --- at least on the Edge-IIoTset benchmark --- and the key is making the
two components work together intelligently rather than simply running both in
parallel.

\noindent The LightGBM classifier proved to be an excellent choice for known
attack classification. Its 99.99\% accuracy on ten attack categories is not
just a high number in a table; it reflects the fact that the packet-level
features in IoT traffic carry very clear signatures for these attacks once
the data is cleaned and properly encoded. The feature importance analysis backs
this up: a relatively small number of TCP header fields are doing most of the
work, and the model learned that quickly without needing an exhaustive
hyperparameter search.

\noindent The Autoencoder surprised us a little. We expected it to be useful but
also expected more false positives given how diverse normal IoT traffic can be.
Achieving 100\% attack recall while keeping the false positive rate at around 5\%
was better than our initial estimates, and the reconstruction error distribution
plot made it clear that the separation between normal and attack traffic in the
learned latent space is genuine, not marginal.

\noindent The Fusion Model brought both components together in a way that made
practical sense. By routing high-confidence decisions through LightGBM and
deferring uncertain ones to the Autoencoder, we avoided the Autoencoder's false
positive problem on easy samples while still getting its zero-day coverage on
harder ones. The final accuracy of 96.82\% on five never-before-seen attack
categories --- compared to 49\% for a standalone Isolation Forest baseline ---
shows that the 47.82 percentage-point improvement is real and not an artefact
of how we set up the evaluation.

\noindent There are genuine limitations here that we have not glossed over:
the system does not name zero-day attacks, it evaluates traffic sample-by-sample
with no temporal context, and the anomaly threshold will need updating as
deployments age. These are solvable problems, not fundamental flaws, and we have
outlined concrete approaches for each in Chapter~6.

\noindent Stepping back, what we found most interesting was not any individual
accuracy number but the structural lesson: supervised and unsupervised methods
are not competing approaches to the same problem. They are solving different
sub-problems, and combining them intelligently is what makes the whole system
stronger than either part. That is a principle that should generalise well beyond
this specific dataset and attack taxonomy.
